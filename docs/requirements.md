## 1. 개요

### 1.1 배경
매년 가족 여행을 다니지만 별도로 기록하지 않아 언제·어디로 갔는지 기억에 남지 않는다.
사진첩을 직접 정리해 블로그로 만드는 작업은 시간이 많이 들어 결국 하지 않게 된다.

### 1.2 목적
**기기 사진첩**의 사진을 **자동으로 분류**하여 여행별로 묶고, **장소·시간을 기록**하며,
**AI가 잘 나온 사진만 선별**해 여행 기록(타임라인)을 자동 생성한다.

### 1.3 한 줄 정의
> "사진만 찍어두면, 여행 기록은 알아서 만들어지는 서비스"

### 1.4 핵심 가치
- **자동화**: 사용자의 수작업을 0에 가깝게
- **회상**: 언제·어디 갔는지 한눈에 보이는 연도별 타임라인
- **선별**: 수백 장 중 잘 나온 사진만 골라 부담 없이 감상

## 2. 범위 (Scope)

### 2.1 포함 (In Scope) — MVP
- 기기 사진첩 접근(권한) 및 사진 메타데이터 수집
- 사진의 시간(촬영일)·위치(EXIF GPS) 기반 여행 자동 그룹핑
- 위치 좌표 → 지명(국가/도시/명소) 역지오코딩
- AI 기반 사진 품질 평가 + 유사 사진 중복 제거 → 베스트 컷 선별
- 여행별 앱 내 타임라인 뷰 (연도 → 여행 → 사진)
- 여행 제목/메모 수동 편집

### 2.2 제외 (Out of Scope) — 차기
- 외부 공개용 블로그 페이지 발행
- 가족 구성원 간 공유/협업 편집
- 영상(비디오) 처리
- 자동 글(여행 후기) 생성
- 구글 포토/iCloud 등 클라우드 소스 연동 (기기 사진첩으로 충분)

### 2.3 가정 및 제약
- 사진은 **기기 사진첩에 존재**해야 동작한다. (클라우드 전용 보관 사진은 기기에 내려받아야 함)
- 위치 정보가 없는 사진은 시간 기준으로만 그룹핑된다(지명 미표시 가능).
- iOS는 "제한적 접근" 선택 시 허용된 사진만 처리된다 → 온보딩에서 "전체 허용" 안내 필요.
- Android는 `ACCESS_MEDIA_LOCATION` 권한이 있어야 EXIF GPS를 얻을 수 있다.
- AI 비전 모델 호출은 **비용이 발생**하므로 처리량/캐싱 전략이 필요하다.
- 인프라: **DB/백엔드 = Supabase**, 앱 빌드/배포 = Expo(EAS).

## 5. 기능 요구사항 (Functional Requirements)

### FR-1. 계정 및 인증
- FR-1.1 사용자는 앱에 로그인할 수 있다. (Supabase Auth — 이메일 또는 소셜)
- FR-1.2 로그인은 여행 기록을 클라우드에 저장·동기화하기 위한 용도다. (사진 접근 자체는 기기 권한)
- FR-1.3 사용자는 계정을 탈퇴하고 데이터를 삭제할 수 있다.

### FR-2. 사진첩 접근 및 메타데이터 수집
- FR-2.1 시스템은 사진첩 접근 권한을 요청한다. (iOS: 사진 접근 / Android: READ_MEDIA_IMAGES + ACCESS_MEDIA_LOCATION)
- FR-2.2 시스템은 사진을 enumeration하여 촬영시각·GPS 위치·해상도·카메라 정보를 수집한다.
- FR-2.3 최초 1회 전체 스캔 + 이후 신규 사진 증분 스캔 방식으로 동작한다.
- FR-2.4 원본 이미지는 서버에 업로드하지 않고 기기에 유지한다. 서버에는 메타데이터와 (선택)썸네일만 저장한다.
- FR-2.5 iOS 제한적 접근 시 `presentPermissionsPicker()`로 사진 추가 허용을 유도한다.

### FR-3. 여행 자동 그룹핑
- FR-3.1 시스템은 촬영 시간의 연속성(예: 연속 N일 이내) + 위치 군집을 기준으로 사진을 하나의 "여행"으로 묶는다.
- FR-3.2 위치 좌표를 지명(국가/도시/명소)으로 변환한다.
- FR-3.3 여행 기간(시작일~종료일)과 대표 장소를 자동 산출한다.
- FR-3.4 GPS가 없는 사진은 시간 기준으로 보조 그룹핑한다.
- FR-3.5 사용자는 자동 그룹핑 결과를 합치거나 분리할 수 있다.

### FR-4. AI 사진 선별
- FR-4.1 시스템은 각 사진의 품질 점수(흐림/노출/구도 등)를 평가한다.
- FR-4.2 시각적으로 유사한 사진을 군집화하고, 군집별 베스트 컷 1장을 선택한다.
- FR-4.3 여행별 대표 커버 사진을 자동 지정한다.
- FR-4.4 사용자는 선별 결과를 보정(추가/제외/대표 변경)할 수 있다.

### FR-5. 타임라인 뷰
- FR-5.1 사용자는 연도별로 여행 목록을 볼 수 있다.
- FR-5.2 여행 카드에 기간·장소·커버 사진·사진 수가 표시된다.
- FR-5.3 여행 상세에서 선별된 사진을 시간/장소 순으로 본다.
- FR-5.4 사용자는 여행 제목과 메모를 편집할 수 있다.

### FR-6. 처리 상태
- FR-6.1 스캔/분석 진행 상태(진행률, 완료/오류)를 표시한다.
- FR-6.2 대량 처리는 백그라운드로 수행하며 앱 재실행 시 이어서 진행한다.

## 6. 비기능 요구사항 (Non-Functional Requirements)

| 분류 | 요구사항 |
|------|----------|
| 성능 | 사진 1,000장 최초 분석을 백그라운드로 처리, 진행률 노출. 메타데이터 수집은 온디바이스로 빠르게 |
| 비용 | AI 비전 호출 결과 캐싱, 동일 사진 재분석 방지, 무료 티어 한도 내 설계 |
| 보안/프라이버시 | 사진 원본 기기 보관(서버 비업로드), 본인만 접근(Supabase RLS), 최소 권한 |
| 가용성 | 온디바이스 처리 우선으로 오프라인에서도 메타 수집 가능, 서버 동기화는 재시도 |
| 확장성 | 그룹핑·선별 로직을 잡(queue) 단위로 분리 |
| 사용성 | 모바일 우선, 핵심 동작 3탭 이내, 권한 온보딩 친절 안내 |
| 호환성 | iOS / Android 양 플랫폼, 기기별 EXIF 편차 대응 |

## 7. 기술 아키텍처 (개략, v0.2)

```
[네이티브 앱 (React Native + Expo)]
   ├─ expo-media-library: 사진 enumeration + creationTime + location(GPS) + EXIF
   ├─ 온디바이스 메타데이터 수집
   ├─ 그룹핑 엔진 (시간·위치 군집)  ── 온디바이스 또는 서버
   ├─ 타임라인 UI
   └─ 원본 이미지: 기기에 유지
        │  (메타데이터/여행/선별결과만 전송)
        ▼
[Supabase]
   ├─ Auth (사용자)
   ├─ Postgres (사진 메타·여행·선별 결과) + RLS
   ├─ Edge Functions (역지오코딩 프록시, AI 선별 호출, 무거운 그룹핑)
   └─ Storage (썸네일, 선택)
        │
        ▼
[외부 API]
   ├─ 역지오코딩 (좌표 → 지명)
   └─ AI 비전 모델 (품질/유사도)
```

### 7.1 외부 의존성 (확인 필요)
- **expo-media-library**: 기기별 EXIF/GPS 출력 편차 — *실기기 PoC로 1차 확인 완료, 출력률 추가 검증 권장*
- **역지오코딩 API**: Google Geocoding / Nominatim 등 비용·정책 비교 (서버 프록시 권장)
- **AI 비전 모델**: 품질·유사도 평가용 모델 선정 (호출 비용이 핵심 변수)

## 9. 사용자 플로우 (핵심, v0.2)

1. 앱 설치 → 로그인 → **사진첩 접근 권한 허용(전체 허용 권장)**
2. "사진 정리 시작" → 백그라운드 스캔 (진행률 표시)
3. 자동 그룹핑 + 지명 변환 + AI 선별 수행
4. 타임라인에 "2025 제주 가족여행 (3일, 사진 24장)" 형태로 자동 생성
5. 사용자가 제목/메모만 살짝 손보면 완성
